The model call

An agent calling a foreign-hosted model is a cross-border transfer of whatever data is in that prompt. Why one routing rule can't cover this

Five jurisdictions. Five different tests for the same act.

Not five strictness levels of one rule — five structurally different mechanisms. A compliance answer built for one doesn't transfer to the next by analogy.

EU
Prohibition, with enumerated exceptions
GDPR Chapter V bans a transfer by default unless an adequacy decision, a Standard Contractual Clause, or a Binding Corporate Rule applies. No exception, no transfer.
UK
Same shape, relaxed bar
A destination only has to be "not materially lower" than the UK's own standard, not a structural match — the 2026 ICO guidance's own words.
TH
A whitelist with nothing on it yet
Thailand's PDPA permits transfer to an "adequate" country — except no adequacy list has been published, so every transfer needs its own signed contract instead.
SG
A standards test, not a country test
Singapore's PDPA asks whether the specific recipient provides "comparable" protection — the same destination country can pass for one recipient and fail for another.
IN
Permission, with a blacklist underneath
India's DPDP Act permits transfer everywhere except a notified restricted list — currently empty — but sectoral rules (RBI, SEBI, IRDAI) still bind underneath it.

A real, fail-closed answer

Two composable policies. No exceptions granted quietly.

ResidencyPolicy answers where — and records the transfer mechanism, not just the outcome. RetentionPolicy answers how long, zero-data-retention included. Both wrap any ModelPolicy, both fail closed on missing context, neither ever falls through to a silent default deployment. 

# jurisdiction -> {deployment: transfer mechanism}
policy = ResidencyPolicy(
    AllowlistModelPolicy(routes),
    allowed_by_jurisdiction={
        "EU": {"eu-deployment": "adequacy_decision"},
        "IN": {"in-deployment": "dpdp_section_16"},
    },
)
policy = RetentionPolicy(policy, deployment_retention_days={"eu-deployment": 0})
guard = ModelGuard(policy=policy)

model = guard.resolve(alice, "default-chat-model", context={"jurisdiction": "EU"})
# -> "eu-deployment", audited as:
# "resolved 'default-chat-model' -> 'eu-deployment' for jurisdiction
#  'EU' via 'adequacy_decision'"

guard.resolve(alice, "default-chat-model", context={"jurisdiction": "TH"})
# -> raises ModelCallDenied. Fails closed — no silent fall-through.

One library, one audit trail

What else Marshal already enforces.

ART. I

Retrieval — RetrievalGuard

Document/field-level access control on any retriever, with redaction built in — an ACL decision and a content-sensitivity decision are different questions, and Marshal answers both.

ART. II

Tool calls — ToolGuard

Risk-tiered allow/deny/require-approval on every tool call, with argument redaction. A denied or declined call raises loudly — never a silent no-op.

ART. III

Model routing — ModelGuard

Resolves a logical model name per principal, with governed fallback chains, per-principal budgets, and now jurisdiction and retention as first-class routing inputs.

ART. IV

Content — marshal_ai.sensitive

Deterministic, regex-based secret/PII detection across every surface — not "is this allowed" but "does the payload itself contain something that shouldn't be here." Findings are recorded as counts, never the matched text.

Marshal is MIT-licensed, has zero required dependencies, and is not yet published to PyPI — clone the repo and install locally. Every decision here is enforced deterministically; none of it is legal advice.
